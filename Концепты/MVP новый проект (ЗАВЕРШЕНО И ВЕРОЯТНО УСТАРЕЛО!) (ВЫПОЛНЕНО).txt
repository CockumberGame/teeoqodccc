Составление текстов

Create a scalable MVP web game inspired by Reigns-style decision gameplay.

GENERAL IDEA
The game is a narrative card decision roguelite.
Each turn the player receives a card with two choices (left/right).
Choices affect player stats and may trigger events.

The project must be built with clean architecture and modular design so that the game can scale to hundreds of cards and future mechanics.

TECH STACK
Frontend:

JavaScript

React

Vite

Tailwind CSS

Backend / persistence:

Supabase (for future player accounts and saves)

The MVP must run fully in browser even without backend.

PROJECT GOALS

clean architecture

modular systems

cards defined as data (JSON)

extendable effect system

support future content expansion

IMPORTANT:
Game logic must NOT be embedded inside cards.
Cards must only contain declarative data.
All gameplay logic must be handled by systems.

ARCHITECTURE

Use a modular structure:

/src
/engine
GameEngine.js
RunManager.js
EffectEngine.js
CardEngine.js
ConditionEngine.js

/systems
PlayerSystem.js
DeckSystem.js
UpgradeSystem.js
FlagSystem.js
SaveSystem.js

/data
cards.json
upgrades.json

/ui
GameScreen.jsx
CardView.jsx
StatsPanel.jsx
BaseScreen.jsx

/styles

main.jsx

start new run

PLAYER STATS

Use a minimal but expandable stat system:

hp
stamina
gold
xp
level
morale

Stats must be stored in a player state object.

CARD SYSTEM

Cards must be defined as JSON data.

Example card structure:

{
"id": "forest_bandits",
"text": "Bandits block your path.",
"tags": ["enemy","forest"],
"weight": 5,
"cooldown": 3,
"choices": [
{
"text": "Fight",
"effects": [
{ "type": "damage", "value": 5 },
{ "type": "gain_gold", "value": 10 }
]
},
{
"text": "Run",

{ "type": "lose_stamina", "value": 3 }
]
}

}

Cards may contain:

tags

but must NOT contain logic.

EFFECT SYSTEM

Create a centralized EffectEngine.

Effects are defined as:

damage
heal
gain_gold
lose_gold
gain_xp
add_flag
remove_flag
add_item
trigger_event

EffectEngine must interpret effect objects and update the player state.

FLAG SYSTEM

Implement a flag system to store story decisions.

Example:

set_flag: "saved_bandit"

Future cards may contain conditions requiring flags.

CONDITION SYSTEM

Cards may include appearance conditions:

minimum level
minimum gold
required flag
missing flag

DECK SYSTEM

The deck must support:

weighted random selection
tag filtering
cooldowns preventing immediate repeats

MVP CONTENT

Generate initial content:

20 cards
3 enemy cards
5 event cards
2 treasure cards
2 merchant cards
8 neutral cards

Also include:

3 upgrades

example upgrades:

increase max hp
increase stamina
increase luck (future stat)

UI DESIGN

Minimalist UI.

Requirements:

flat design
layered cards with soft shadows
no gradients
avoid dark blue tones

Color palette:

silver tones for base UI
gold accents for important elements

Use pixel-style fonts.

INTERFACE

Main layout:

top bar → player stats
center → card
bottom → choice buttons

The UI must support swipe gestures later, but MVP can use buttons.

SCALABILITY

The architecture must allow:

adding cards without touching code
adding new effects easily
adding new stats
adding new UI modules

Provide clear comments explaining each system.

OUTPUT

Create a working MVP that can run locally with:

npm install
npm run dev

Отличный вопрос. У проектов, которые пишутся через ИИ (Codex или подобные инструменты), есть несколько типичных архитектурных ловушек. Они почти всегда появляются через пару недель разработки, когда начинаешь расширять игру. Разберём самые важные. 🔎

1. «God Object» — главный файл-монстр

Самая частая ошибка: вся игра постепенно съезжает в один файл вроде:

game.js

Внутри оказывается всё:

логика игрока
карточки
эффекты
UI
сохранения
колода
апгрейды

Через 2–3 недели файл становится 2000+ строк, и менять что-то становится опасно.

Правильное решение

Каждая система — отдельный модуль.

engine/
  GameEngine
  CardEngine
  EffectEngine
systems/
  PlayerSystem
  DeckSystem
  UpgradeSystem
  FlagSystem
ui/
  CardView
  GameScreen
  StatsPanel

Главный движок только оркестрирует системы.

Пример:

GameEngine.update() {
  deck.draw()
  cardEngine.presentCard()
}
2. Логика внутри карточек

Это вторая самая частая катастрофа.

ИИ иногда начинает генерировать карточки типа:

if(player.hp < 5){
  player.hp += 10
}

И внезапно каждая карточка становится маленьким скриптом.

Через 100 карт:

балансировать невозможно

генерировать карты невозможно

код становится неуправляемым

Правильный подход

Карточки = чистые данные

{
 "effects":[
   {"type":"heal","value":10}
 ]
}

А вся логика живёт в Effect Engine.

3. Статы жёстко прошиты в код

Очень частая проблема.

ИИ пишет:

player.hp
player.gold
player.stamina

А потом ты хочешь добавить:

morale
sanity
luck
faith

И приходится переписывать половину игры.

Правильный подход

Статы должны быть динамической системой.

Пример:

player.stats = {
  hp: 20,
  stamina: 10,
  gold: 5
}

Эффекты работают так:

player.stats[effect.stat] += effect.value

Теперь можно добавить новый стат без изменения движка.

4. Колода без менеджера состояния

Новички делают так:

cards[random]
cards[random]
cards[random]

В итоге:

бандиты
бандиты
бандиты
бандиты
Нужен Deck Manager

Он хранит:

активную колоду
cooldown карт
последние карты
веса

И решает какая карта может появиться.

5. Отсутствие Event Bus

Это уже более продвинутый момент.

Без него системы начинают напрямую вызывать друг друга:

CardEngine → PlayerSystem → UI → SaveSystem

И всё переплетается.

Лучше использовать событие
cardChosen
statChanged
runEnded
upgradeBought

Пример:

eventBus.emit("cardChosen", choice)

Теперь системы могут слушать события, но не зависеть друг от друга.

6. Контент смешан с кодом

Если карточки лежат внутри:

cards.js

это проблема.

Нужно хранить их как контент.

data/
  cards/
  events/
  upgrades/

Тогда можно:

генерировать их через AI

редактировать без кода

добавлять контент-паки

7. Отсутствие Run State

Очень частая проблема roguelite-игр.

ИИ иногда хранит всё в player.

Но run и метапрогресс — разные вещи.

Правильная модель:

Player (постоянный прогресс)

Run (текущий забег)

Пример:

player.gold_total
player.upgrades

run.hp
run.gold
run.flags

После смерти run удаляется.

8. UI напрямую управляет игрой

Очень плохой паттерн:

button.onclick = () => {
  player.hp -= 5
}

UI не должен менять состояние игры.

Правильно:

UI → GameEngine → Systems
9. Отсутствие версии контента

Когда у тебя будет 200+ карт, появится проблема:

игроки могут иметь старые сохранения.

Лучше сразу добавить:

content_version

Пример:

data/version.json
10. Нет системы тегов

Без тегов карточная система быстро ломается.

Нужно сразу:

tags:
enemy
event
merchant
treasure
curse
story

DeckManager сможет строить колоду:

40% enemy
30% event
20% neutral
10% rare
Архитектура, которая хорошо работает для таких игр

Финальная схема:

GameEngine
  │
  ├ CardEngine
  ├ EffectEngine
  ├ ConditionEngine
  ├ DeckSystem
  ├ PlayerSystem
  ├ RunManager
  ├ FlagSystem
  └ UpgradeSystem

UI отдельно.

Контент отдельно.

Ещё одна очень полезная вещь для карточных игр

Система Simulation Mode.

Это маленький скрипт, который запускает 1000 автоматических забегов.

Он позволяет увидеть:

среднюю длину игры
скорость смерти
экономику золота

Такие симуляции используют даже крупные студии.

Она называется data-driven design, и она позволяет расширять игру почти бесконечно без переписывания движка.
